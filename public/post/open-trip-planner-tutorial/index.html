OpenTripPlanner Tutorial (for real beginners!)

Jan 1, 0001 4 min read

At work, I was recently tasked with routing several hundred thousand trips through the Twin Cities transit network. With origin, destination, and departure time provided, my goal was to determine how a trip could be made by walking and taking Metro Transit bus or light rail. I’ve been hearing about OpenTripPlanner for a few months and knew this was probably the tool I wanted. Unfortunately, I found the homepage less than instructive and the basic tutorial a bit lacking for someone with no background knowledge of OpenStreetMaps or GTFS feeds. Here, I’ve tried to compile the resources I wish I’d found when I looked up a tutorial!

What is OpenTripPlanner?

OpenTripPlanner is a “multimodal trip planner”. In other words, it allows us to calculate travel times and create trip itineraries for most modes of transportation – walking, biking, driving, and public transit where it exists! Maybe the most common (or well-publicized) use of OpenTripPlanner is creating isochrones. Isochrones are travel time maps. They show everywhere you can reach from a specific starting location in various amounts of time. For example, here’s an isochrone showing walk times from my old dorm at Macalester (more on how I created this later!).

Isochrones are a cool tool used by places like the Accessibility Observatory to think about who can get where. My goal with OpenTripPlanner was a little different. I had data about trips taken via car and wanted to determine which could be made in a comparable amount of time on public transit.

OpenTripPlanner requires two main inputs to create trip plans and isochrones. First, we need a basemap. This usally comes from OpenStreetMaps and tells OpenTripPlanner the location of geographic features like roads, sidwealks, and bikeways. If we’re interested in public transit (which I usually am!), we also need a transit schedule in the form of a GTFS feed. Depending on your specific application, you may want a few optional inputs, like a Digital Elevation Model for identifying hills (great for planning bike trips!).

GTFS

GTFS stands for General Transit Feed Specification. Originally, the G stood for ‘Google’ but GTFS is increasingly being considered the standard way to share transit schedules.

GTFS is actually just a collection of .txt files. Each file contains a different component of the transit schedule. GTFS feeds are almost always provided as a zipped file so none of the components get lost. If you unzip a GTFS feed, you’ll see something like this:

Setting up OpenTripPlanner

Prerequisites

OpenTripPlanner is written in Java, so we need to have that installed before we can get up and running. You can check which Java version (if any) you have installed by typing java --version in the terminal. If you see Java 1.8 or later, you’re good to go. Otherwise, you have two options. You can follow the installation instructions on the Java site, or use brew cask install java if you have homebrew. More on installing with brew here. To use command line Java prompts, you’ll also need to install a Java Developer Kit (this should be done if you use Homebrew for installation).

Next, we can install OpenTripPlanner. This is probably the easiest part of the process! First, go to the OTP directory. You’ll want to click on the most recent version and download the file that ends in ‘shaded.jar’. As I’m writing this, the most recent release is 1.4.0.

I put this file in a folder called ‘otp’ at /Users/ravenmcknight/otp. This is the directory we’ll work from whenever we run OpenTripPlanner.

The last pieces we need are some tools to make running OpenTripPlanner from the terminal easier. I installed each of these using Homebrew:

  • brew install wget to download files directly from URLs
  • brew install osmfilter to handle OpenStreetMap files. After you run this, use which osmconvert to make sure the tool we need was installed properly. The output should be something like ‘/usr/local/bin/osmconvert’.

Data

There are a few ways to get a basemap from OpenStreetMaps.

  • downloading a whole state (osmconvert) If you use this method, it’s important to crop your basemap down. The larger your basemap, the longer OpenTripPlanner will take to render. Unless it’s necessary, I wouldn’t recommend downloading the whole state.

  • HOT maps (the issue is that I needed a shapefile with no interior boundaries)

  • Is there an API?

Ideally, getting a GTFS file should be straightforward. Not all GTFS feeds are created equal, though, and some of them will have errors
